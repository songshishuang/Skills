# 决策记录 (Architecture Decision Records, ADR)

本目录沉淀 **{{PROJECT_NAME}}** 的产品 / 技术 / 商业重要决策。**回答的不是"怎么做"，而是"为什么这么决定的"**——半年后回看仍能复盘。

## 何时写 ADR

判断标准：**「这个决策如果反悔，会影响 ≥ 2 周工时或 ≥ 2 个角色」**

- ✅ 技术选型（用 / 不用某框架）
- ✅ 架构选择（多租户隔离 / 数据存储方案）
- ✅ 产品定位重大调整
- ✅ 商业模式重要变化
- ❌ UI 微调
- ❌ 临时性方案

## 文件命名

`ADR-NNN-{slug}.md`

- `NNN`：三位顺序编号（001, 002, ...）
- `slug`：英文小写连字符

例：`ADR-001-why-not-langchain.md`

## 标准结构

参考 [`ADR-template.md`](./ADR-template.md)：

1. 状态 / 决策日期 / 决策人 / 相关人
2. 上下文 (Context)
3. 候选方案 (Options Considered) — 至少 2 个
4. 决策 (Decision)
5. 后果 (Consequences) — 正面 / 负面 / 长期
6. 验证 (Verification) — 怎么知道对错、什么时间点回看

## 维护约定

- **不修改已生效的 ADR** — 决策变更时新建一个标注「替代 ADR-XXX」
- **每季度复盘** — 查标了"什么时候回看"的 ADR
- **失败的决策也保留** — 状态改"已废弃 / 已被替代"，不删除——后人能学到避坑经验
