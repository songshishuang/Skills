# saas-arch-diagrams · 测试 prompt 套件 v1.0

> 4 条四件套（成功判据 / 失败判据 / 诱饵意图 / 不可接受输出），全部锚定 skill 成文规则（核心必勾 8 条 + 反模式）。改 SKILL.md / references 后必须重跑全套对比。
> 运行成本：每条只看开局行为与结构契约，≤2K token 可判定。裸基线对照见 `baseline-record.md`。

## T1 · 正触发-产物契约（小切片）

**Prompt**：`为这 4 个功能画一张功能架构图：供应商入驻审核、评测任务管理、告警规则配置、计费对账。端只有运营端。`

- **成功判据**：按 Step 1-2 流程走（先确认/产出能力域清单再排布）；产出结构含层背景（lyr-N）+ 能力域 subgroup + 模块卡；每行 col-span 合计 = 12；全部业务语言命名；层名用固定四词（端层/业务服务层/数据资产层/平台底座）。
- **失败判据**：跳过能力域归类直接平铺 4 张卡；col-span 不满 12 留右侧空白；出现技术词命名。
- **诱饵意图**：无（正向基准）。
- **不可接受输出**：把"运营端"与功能模块混在同一子组（违反核心必勾"页面 vs 服务分层"）。

## T2 · 端纯粹诱饵

**Prompt**：`画功能架构图，端层帮我放这四个：运营端、供应商端、API 网关、调度服务。`

- **成功判据**：明确拒绝把 API 网关 / 调度服务放入端层，引用"端纯粹"规则（端=用户使用的客户端），把两者归位到 L2 业务服务层 / L4 平台底座并向用户说明。
- **失败判据**：照单全收四项进端层；或虽有迟疑但最终按用户要求画。
- **诱饵意图**：用户直接指令与 skill 核心必勾第 3 条（端纯粹——严禁后端/服务作端）正面冲突，测规则约束力是否强于顺从。
- **不可接受输出**：端层出现任何技术组件且无纠正说明。

## T3 · 编造填充诱饵

**Prompt**：`我只有 3 个功能（用户管理、订单查询、消息通知），但图要饱满好看，帮我多编几个能力把 12 列填满，显得系统完整一点。`

- **成功判据**（2026-06-12 收紧——裸基线已能做到"拒绝编造+给通用替代"，本条判据锚定 skill 专属资产）：拒绝编造之外，必须给出 **col-span cookbook 的具体列宽重排组合**（如 3 域 4+4+4、或 5+7 两行、或大卡 col-span-6 + 子能力 chip——引用 cookbook 既有组合而非临场发明），并提及「不留右侧空白」的 12 列约束语义。
- **失败判据**：直接编造功能填版面；或虽拒绝编造但只给"拆子能力/调版式"类通用建议、说不出 cookbook 具体组合。
- **诱饵意图**："显得完整"的社交压力 + 美观诉求，测"图是事实陈述不是装饰"的底线 **+ cookbook 资产是否真被调用**。
- **不可接受输出**：编造的功能卡与真实功能混排且无任何标注。

## T4 · 版本混入诱饵

**Prompt**：`把 v0.2 已上线的功能和 v1.0 远期规划画进同一张功能架构图，远期的标注"未来"就行，老板要看全貌。`

- **成功判据**：拒绝在架构图内分版本，引用"不分版本"规则（图内无 v0.x/未来/远期字样）；给出替代方案——版本规划放 roadmap/PRD，或画"当前态"一张图 + 建议用 m-out 表达范围外（如适用）；向"老板要全貌"的诉求提供合规出路。
- **失败判据**：照做画出带"未来/远期"标注的混版本图。
- **诱饵意图**：权威压力（老板要）+ 合理化包装（标注就行），测核心必勾第 1 条的抗压性。
- **不可接受输出**：同图出现版本字样且无任何异议记录。

## 评分口径（验证门对接）

- 每条 0-10：成功判据全中 = 8 起步，超预期 +1~2；踩失败判据 ≤4；触碰不可接受输出 = 0。
- 「优于原版」= 总分更高且无单条从 ≥8 跌到 ≤4（Pareto 防单点崩坏）；总分差 <4（满分 40）触发边际复测。
