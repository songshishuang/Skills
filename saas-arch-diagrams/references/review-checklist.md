# 架构图评审 Checklist（v2.0 分层 · 不再"全部必勾"）

每次提交前对照下表逐条勾。**核心必勾**任意一条没勾代表会被读图者抓出来返工；**场景必勾**视项目特点决定是否适用；**推荐**漏勾不算错。

> 旧版 21 条全部"必勾"在小项目 / 单页架构图场景过度约束。v2.0 按场景分层（参考 prd-writer v2.3 的 DoD 分层）。

---

## 🔴 核心必勾（任何架构图都要 · 8 条）

> 这些是 SaaS 多端架构图的底线，违反任意一条 = 不可用。

- [ ] **不分版本**：图内无"v0.1 / 未来 / 远期"等字样。版本规划放 roadmap / PRD
- [ ] **不省略合并**：所有功能模块独立成卡，不出现"X-Y 高阶节点 5 个合并"
- [ ] **端纯粹**：只有用户使用的客户端（运营端 / 供应商端 / 客户端 / 管理员端等）。**严禁**"后端 / 业务服务 / 数据层"等技术词作端
- [ ] **页面 vs 服务分层**：UI 入口属 L1 端层，业务能力属 L2 服务层，绝不混在一个子组
- [ ] **col-span = 12**：每行子组 col-span 加起来 = 12（grid 12 列），不留右侧空白
- [ ] **描述用业务语言**：避免「中间件 / 微服务 / 后端 / API 网关」等技术词
- [ ] **卡片标题 = 用户能理解的功能名**（如「评测任务列表」而非 `eval_run_list_view`）
- [ ] **层名固定用词**：「端层 / 业务服务层 / 数据资产层 / 平台底座」

---

## 🟡 场景必勾（视项目特点 · 8 条）

> 触发条件不满足时可跳过；满足时必勾。

- [ ] **能力域内分类**（≥ 2 个能力域时）：每个域内功能按子能力（A1/A2/A3...）分组，不平铺
- [ ] **数据资产 vs 管理工具拆开**（涉及数据产品 / 评测集 / 知识库时）：数据本身和管理 UI 是两个维度
- [ ] **横切关注点独立**（涉及治理 / 安全 / 计量 等跨域能力时）：不塞进具体域，用横切表达
- [ ] **不在本平台范围用 dotted**（涉及下游 / 外部依赖时）：dotted 置灰 + line-through（class `m-out`）
- [ ] **L2 / L3 嵌套容器区分**（3 级嵌套时）：L2 用实色背景 + 实线边框；L3 用虚线轻量框
- [ ] **与产品架构图能力域命名一致**（同时有产品架构图 + 功能架构图时）：两图命名口径必须对齐
- [ ] **与 PRD 内功能命名一致**（同时有 PRD 时）：「评测任务列表」全局统一
- [ ] **与原型实际入口对应**（已有原型时）：卡片名 == 原型页面标题

---

## 🟢 推荐（可选 · 漏勾不算错 · 5 条）

> 提升可读性 / 评审友好度，但不是底线。

- [ ] 一行内 col-span 高度差不超过 2 倍；高度差大时改让小子组单独成行
- [ ] 卡片副标 ≤ 30 字 · 一句话讲清楚做什么，不堆术语
- [ ] 子能力名称简短（如「入驻准入」≤ 8 字）
- [ ] 每个端 / 能力域 / 子能力 有清晰 ID 编码（A1 / B2 / C3 / D4），方便引用
- [ ] 图例完整：能力域 chip + 端类型 + 状态色 + 读图说明

---

## 🎨 视觉约定（项目特化 · 仅作示例 · 非必填）

> 以下色彩 / chip 命名是某 LLM 服务聚合平台项目的实战约定，**其他项目应当按自家品牌 / 业务自定义**，不要硬套。

- 卡片左 3px accent 按所属能力域着色（**示例**：A 紫 / B 琥珀 / C 蓝 / D 玫红 / X 灰）
- 右上角能力域 chip（**示例**：A / B / C / D / X · 含义见各项目定义）
- 子组容器：L2 实色背景 + 实线边框；L3 虚线轻量框
- 顶部统计区 / Topbar 跨图导航 / 底部规模统计 —— 大型 / 多图项目可加，小项目可不加

**Anti-pattern**：照搬上述色彩 / 命名当作本项目的"必填规则"——5 色 A-X 是项目特化，强加给其他项目是反模式。请在 v2.x Self-Evolving 反向简化清单中检查。

## 自检流程

每次改完执行：

1. 截图整张图，对照核心必勾 8 条 + 适用的场景必勾，逐条勾掉
2. 想象自己是「跨团队同事第一次看这张图」，能否在 30 秒内理解：
   - 我们是谁的子系统？
   - 谁在用？
   - 有哪几个能力域？
   - 哪些不在我们做？
3. 想象自己是「PM 做范围判定」，能否在 1 分钟内：
   - 找到某个具体功能在哪一层 / 域 / 子能力？
   - 判断这个功能依赖哪些其他模块？

无法做到 → 继续优化层次和命名（**不是**继续加 checklist 项）。
