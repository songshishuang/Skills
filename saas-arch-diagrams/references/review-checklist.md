# 架构图评审 Checklist

每次提交前对照下表，逐条勾掉。任何一条没勾代表会被读图者抓出来返工。

## 内容合理性（最容易被打回）

- [ ] **不分版本**：图内无"v0.1 / 未来 / 远期" 等字样。版本规划放 roadmap / PRD。
- [ ] **不省略合并**：所有功能模块独立成卡，不出现"X-Y 高阶节点 5 个合并"。
- [ ] **端纯粹**：只有用户使用的客户端（运营端 / 供应商端 / 第三方平台 / 客户端 / 管理员端）。**严禁** "后端 / 业务服务 / 数据层" 等技术词作端。
- [ ] **页面 vs 服务分层**：UI 入口属 L1 端层，业务能力属 L2 服务层，绝不混在一个子组。
- [ ] **能力域内分类**：每个域内功能按子能力（A1/A2/A3...）分组，不平铺。
- [ ] **数据资产 vs 管理工具拆开**：评测集数据本身和评测集管理 UI 是两个维度。
- [ ] **横切关注点独立**：治理 / 安全 / 计量等跨域能力不塞进具体域，用横切表达。

## 视觉与布局

- [ ] 每行子组 col-span 加起来 = 12（grid 12 列），不留右侧空白。
- [ ] 一行内 col-span 高度差不超过 2 倍；高度差大时改让小子组单独成行。
- [ ] 每个卡片左 3px accent 按所属能力域着色（A 紫 / B 琥珀 / C 蓝 / D 玫红 / X 灰）。
- [ ] 每个卡片右上角能力域 chip（A/B/C/D/X）。
- [ ] 不在本平台范围的能力用 dotted 置灰 + line-through（class `m-out`）。
- [ ] 图例完整：能力域 chip + 端类型 + 状态色 + 读图说明。
- [ ] 子组容器：L2 用实色背景 + 实线边框；L3 用虚线轻量框。

## 文字与语义

- [ ] 描述用业务语言。避免「中间件 / 微服务 / 后端 / API 网关」等技术词。
- [ ] 卡片标题 = 用户能理解的功能名（如「评测任务列表」而非 `eval_run_list_view`）。
- [ ] 卡片副标 = 一句话讲清楚做什么（≤ 30 字），不堆术语。
- [ ] 子能力名称简短（A1 入驻准入 ≤ 8 字）。
- [ ] 层名 = 「端层 / 业务服务层 / 数据资产层 / 平台底座」固定用词。

## 一致性

- [ ] 与产品架构图（如有）的能力域命名一致：A 供应商生命周期 / B 质量评估 / C 资源 & 商务 / D 风险与治理。
- [ ] 与 PRD 内的功能命名一致（如「评测任务列表」全局统一）。
- [ ] 与原型实际入口对应（卡片名 == 原型页面标题）。

## 演进与维护

- [ ] 每个端 / 能力域 / 子能力 都有清晰 ID 编码（A1 / B2 / C3 / D4），方便引用。
- [ ] 底部统计区：按层数 / 模块数汇总，让规模一目了然。
- [ ] Topbar 提供跨架构图导航（产品架构图 ⇄ 功能架构图 ⇄ 原型）。

## 自检流程

每次改完执行：

1. 截图整张图，对照 checklist 逐条勾掉
2. 想象自己是「跨团队同事第一次看这张图」，能否在 30 秒内理解：
   - 我们是谁的子系统？
   - 谁在用？
   - 有哪几个能力域？
   - 哪些不在我们做？
3. 想象自己是「PM 做范围判定」，能否在 1 分钟内：
   - 找到某个具体功能在哪一层 / 域 / 子能力？
   - 判断这个功能依赖哪些其他模块？

无法做到 → 继续优化层次和命名。
