# 反模式与对照案例

每条都来自 实战项目的真实返工。

---

## 反模式 1：把数据资产和管理工具混在一起

### ❌ 错误版本

```
B3 评测集（8 模块）
├── 评测集管理 console
├── L1 标准公开集
├── L2 行业场景集
├── 性能评测集
├── 评测集导入向导
├── 字段映射 + Metric 配置
├── 基础校验 + 试跑
└── 评测集版本管理
```

### 问题

"评测集管理 console / 导入向导 / 字段映射 / 校验 / 版本管理"是**管理工具**（动作），"L1 / L2 / 性能 / 自定义" 是**数据资产**（名词）。混在一个子组让人看不清"我在管什么"和"我用什么管"。

### ✅ 修正版本

```
B3 评测集 · 数据资产（按层级）
├── L1 标准公开集
├── L2 行业场景集
├── L3 黄金集
├── L4 探针集
├── L5 红蓝对抗集
├── 性能评测集
└── 自定义评测集

B4 评测集 · 管理工具
├── 评测集管理 console
├── 评测集导入向导
├── 自定义评测集上传
├── 性能评测集上传
├── 字段映射 + Metric 配置
├── 基础校验 + 试跑
└── 评测集版本管理
```

## 反模式 2：基线塞进"报告产出"

### ❌ 错误版本

```
B4 报告产出（4）
├── 体检报告 + 审批
├── 行业基线对照库
├── 评级证书生成
└── 赛马看板
```

### 问题

行业基线（model_baseline.json · 6 模型 × 5 集合 = 30 条）是**数据资产**，供报告引用，不是报告本身。评级证书 + 赛马看板是 **评级与对比**类，不是报告。审批面板是 L1 端层。

### ✅ 修正版本

```
B5 评测报告 + 审批 → 体检报告 + 审批
B6 评级与对比 → 评级证书 / 赛马看板
B7 行业基线对照库 → 单独子组（属数据资产视角）
```

## 反模式 3："层 6a / 6b" 这种 a/b 子层

### ❌ 错误版本

```
层 6a · 对象（Model / Endpoint）
层 6b · 底座（沙箱 / Judge / DLP ...）
```

### 问题

a/b 命名表示"同层内的两件不同事"，但实际"评测对象"和"评测底座"是完全不同维度。要么独立成层，要么并入其他层。

### ✅ 修正版本

```
层 6 评测对象（Model / Endpoint · 2 卡）
层 7 评测底座（10 服务）
```

## 反模式 4：评级 / 综合评分 / 行业基线 / 审批面板 全塞一个子组

### ❌ 错误版本

```
③ WHAT · 评估产出（5）
├── 综合评分计算
├── 行业基线对照
├── 风险警示
├── 审批决策面板
└── 评级算法（ABCDE）
```

### 问题

- 综合评分计算 → 评估产出 ✓
- 行业基线对照 → 是「数据资产」（model_baseline 库），不是"产出"
- 风险警示 → 评估产出 ✓
- 审批决策面板 → 是「L1 端层页面」，不是引擎内部
- 评级算法 ABCDE → 是「D 风险治理域」的能力，跨域引用

5 个东西分别属于不同层 / 不同能力域，硬塞一起。

### ✅ 修正版本

```
③ WHAT · 评估产出（3 个产出）
├── 综合评分计算
├── 风险警示信号
└── 报告数据结构

引用其他层（小注脚）：
· 行业基线对照 → 层 7 评测底座 · model_baseline 库
· 审批决策面板 → 层 2 体检报告 + 审批
· 评级算法（ABCDE）→ D 风险治理域
```

## 反模式 5：把"后端 / 评测工程师"列为端

### ❌ 错误版本

```
端层
├── 🛠 运营端
├── 🧪 评测工程师
├── 🏢 供应商端
├── 🖥 后端
└── 🔀 下游聚合平台
```

### 问题

- "评测工程师" = 运营端的内部角色，不是独立的端
- "后端" = 技术实现，不是端

端的定义：用户用什么界面访问平台。**只能是**人面对的客户端。

### ✅ 修正版本

```
L1 端层（用户界面）
├── 🛠 运营端 · admin 内部（评测工程师 / 运营 / 采购等多角色共用）
├── 🏢 供应商端 · vendor portal
└── 🔀 下游聚合平台 · 下游消费方（不在本平台范围）

L2 业务服务层（页面背后的能力 · 无 UI）
├── A 域服务 / B 域服务 / C 域服务 / D 域服务
```

## 反模式 6：col-span 等宽分配导致空白

### ❌ 错误版本

```
D 域：D1(2) + D2(3) + D3(2) + D4(2) + D5(1) + D6(1) = 11 卡
布局：grid-cols-4，每个子组 col-span-3
结果：D5 / D6 只有 1 卡 → 下方留 col-span-9 空白
```

### 问题

固定 col-span-3 时，卡片少的子组无法填满高度。

### ✅ 修正版本

```
D 域两行布局：
行 1: D1(2) col-3 + D2(3) col-9 → 12 ✓
行 2: D3(2) col-3 + D4(2) col-3 + D5(1) col-3 + D6(1) col-3 → 12 ✓
```

按内容动态分配 col-span，使每行加起来 = 12。

## 反模式 7：能力域内平铺

### ❌ 错误版本

```
运营端 · B 域 评测工作（11 页面）
[卡 1] [卡 2] [卡 3] [卡 4] [卡 5] [卡 6] [卡 7] [卡 8] [卡 9] [卡 10] [卡 11]
```

### 问题

11 张卡片并排或换行排列，没有逻辑分组。用户看到这一坨完全不知道哪些是"做评测"、哪些是"管评测集"、哪些是"看结果"。

### ✅ 修正版本

```
运营端 · B 域 评测工作
├── B1 评测任务（4）：任务列表 / 新建 / 流水线 / 报告审批
├── B2 评测集管理（4）：console / 导入 / 自定义 / 性能集
├── B3 评级与对比（2）：评级证书 / 赛马
└── B4 基线维护（1）：行业基线
```

每个能力域内按"子能力"再分一层，4 个子组横向排开（col-4 + col-4 + col-2 + col-2 = 12）。

## 反模式 8：版本视角混入全局图

### ❌ 错误版本

```
B2 节点执行
├── ① 连通性 [v0.1]
├── ② 性能评测 [v0.1]
├── ③ 能力评测 [v0.1]
└── ④-⑧ 高阶节点（远期）
```

### 问题

「远期」「v0.1」标签让图变成"版本规划"而非"全局功能视图"。读者会把它当 roadmap 看。

### ✅ 修正版本

```
B2 评测节点（8 个独立卡）
├── ① 连通性
├── ② 性能评测
├── ③ 能力评测
├── ④ 安全合规
├── ⑤ 计费校准
├── ⑥ 反套壳指纹
├── ⑦ 漂移检测
└── ⑧ 基线快照
```

版本规划放 roadmap.md / PRD，架构图只表达"我们做哪些事"。

## 反模式 9：用"v0.1 节点核心"作为视觉强调

### ❌ 错误版本

```css
.m-core { background: amber }  /* "v0.1 节点核心" */
```

### 问题

"v0.1 节点核心"混了版本和功能两个语义。

### ✅ 修正版本

```css
.m-core { background: amber }  /* 改名"核心节点高亮（①②③）" */
```

视觉强调的是"评测流水线的核心节点"，与版本无关。

## 反模式 10：下游聚合平台职责用蓝色实线（看起来像本平台功能）

### ❌ 错误版本

```
C4 路由（与其他 C1-C3 同色实线）
├── 评级 → 路由权重映射
└── 流量分配 / 限流 / 熔断
```

### 问题

蓝实线 + 同色 chip 让读者以为这些功能本平台做。实际是 下游聚合平台（下游）的职责。

### ✅ 修正版本

```css
.m-out {
  background: #f5f5f4;
  border-style: dotted;
  opacity: .7;
}
.m-out .ct { text-decoration: line-through; color: #78716c; }
```

dotted 边框 + 半透明 + 文字 line-through，明确表示"不在本平台范围"。
