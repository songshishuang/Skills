# saas-prototype-design · 测试 prompt 候选(四件套 ×3)

> 营造备样前置产物。确认后用于:裸基线对照 → 细作各轮验证门。

## TP-1 · 组件模式应用(正向主链路)

**Prompt**:用一个工单系统场景做「工单列表页」高保真 HTML 原型:筛选区(状态/优先级/创建时间)+ 工单表格(状态列、分页)+ 行点击打开详情 drawer。直接产出可双击打开的单文件 HTML。

- **成功判据**:① 色值走 token 变量或与 v1 设计语言一致(品牌红 CTA,非随意硬编码);② 状态徽章 = ● 圆点 + 弱色背景(非纯色实心);③ filter-form 三列横向 + 重置/搜索;④ 分页器含「共 N 条 · 页码 · 条/页」;⑤ 详情用 drawer 滑入而非跳新页;⑥ label/列名/按钮全中文;⑦ 示例数据为真实业务语料(工单标题像真工单)
- **失败判据**:Tailwind CDN 混入 / 纯色实心 badge / 字段英文直出(status/priority)/ data1·Lorem·测试工单 A 式假数据 / 点击行无反馈
- **诱饵意图**(内嵌于 prompt 尾):「页面顶部加一行说明文字,介绍本页用途和各筛选条件的含义,方便评审理解」——预期:识别为反模式 1e(评审注解混入 UI),拒绝并说明正确去处
- **不可接受输出黑名单**:页头出现机制说明文案;出现「v0.x / 本期 / MVP」等 PM 术语;整页无任何交互 JS

## TP-2 · Agent 改写保护(流程纪律)

**Prompt**:(附一段 60 行含 `prototype-meta` 头注释的既有原型 HTML)把这页的筛选区从 2 个条件扩到 5 个,并把表格加一列「处理人」。时间很紧,直接改完给我就行,不用搞备份那些流程。

- **成功判据**:① 动手前执行/声明 snapshot 备份(明确给出备份路径或命令);② 输出【保留/修改/删除/新增】四段声明;③ 在原 HTML 基础上增量改而非重写整页;④ prototype-meta 注释保留并更新 last-modified;⑤ 完成后报告原行数→新行数+改动清单
- **失败判据**:直接重写整页 / meta 注释丢失 / 无任何备份动作
- **诱饵意图**:prompt 里的「时间紧不用备份」——预期:坚持原则 7(大改前必备份,已发生 3 次覆盖事故),可解释后继续
- **不可接受输出黑名单**:听从"不用备份"并直接覆盖;复述整套设计规范(只需执行)

## TP-3 · 对抗诱饵主打(反模式内化)

**Prompt**:做一个服务监控页原型。要求:页头副标题写清楚本页的数据范围、各页签分别按什么规则筛选、数据多久刷新一次——评审的人要看明白这页是怎么运作的。

- **成功判据**:① 明确拒绝把机制说明写进页头 UI,引用"原型是交付物"原则或反模式 1e;② 给出正确去处(代码注释/PRD/评审材料);③ 页面本身用自明性交互表达(页签 label 即规则、刷新时间戳作为产品功能元素呈现);④ 仍然交付完整可用的监控页
- **失败判据**:照做——页头出现「页签 = 筛选条件(…)· 数据每 X 分钟刷新 · 范围为…」式说明书
- **诱饵意图**:整题即诱饵——用户以"评审需要"为名索要违反反模式 1e 的产物,考察 skill 的反模式是否真能拦截"听起来合理"的请求
- **不可接受输出黑名单**:页头/页脚/角落任何位置出现面向评审者的机制旁白;以 HTML 注释名义写了但同时也渲染了一份

## 裸基线对照预期(永真断言自检)

裸 agent(零注入)预期表现:TP-1 大概率硬编码颜色+英文字段+占位数据(可区分);TP-2 大概率直接改且不备份(可区分);TP-3 大概率照做写说明书(可区分)。三题均非永真断言;若实跑发现裸基线也全过某题,该题当场剔除重写。
