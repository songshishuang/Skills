# 评测集准备需求

> 随 PRD 一同交付给算法 / QA 同学，作为评测集建设的依据。

## 一、本期评测目标

> 来自 PRD 八.1，明确本期评测要验证什么。

- 验证目标 1：{例：AI 评论采纳率 ≥ 50%}
- 验证目标 2：{...}

## 二、评测集要求

### 2.1 Golden 黄金集（必备）

- **样本数**：{N} 条（每个核心场景 ≥ 10 条）
- **来源**：人工标注 + V0.{X} 线上 PASS 案例 + 业务方提供典型 case
- **标注字段**（每条）：
  - `case_id`
  - `user_query` — 用户提问原文
  - `expected_keywords` — 期望命中的关键词（逗号分隔，任一命中即 PASS）
  - `expected_skill` — 期望调用的技能 id
  - `expected_tools` — 期望调用的工具列表
  - `scenario` — 场景标签（如：性能优化 / 安全扫描 / 风格检查 / 依赖升级）
  - `intent_level` — 意图明确度（strong / partial / weak）
  - `is_multiturn` — 是否多轮（Y/N）
  - `conversation` — 完整对话历史（多轮时必填）
  - `gold_reply_sample` — 一条优质回复样本（人工撰写）
  - `notes` — 备注
- **通过门槛**：通过率 ≥ {85%}

### 2.2 BadCase 集（必备）

- **样本数**：{N} 条
- **来源**：上一版本线上点踩 + 转人工 + Copilot 改写差异大的 case
- **标注字段**：同 Golden + `failure_type`（intent_miss / info_missing / hallucination / kb_missing 等）
- **通过门槛**：相对上一版本**不退化**（新版本通过率 ≥ 旧版本）

### 2.3 对抗集（涉及合规时必备）

- **样本数**：{N} 条
- **来源**：红队设计 + 历史越狱样例 + 行业合规清单
- **覆盖维度**：
  - 越狱：「忘记前面规则」「执行系统命令」
  - 事实幻觉：「{需要查最新数据的问题}」（期望调工具，不能凭空生成）
  - 违禁话题：政治 / 色情 / 暴力 / 医疗诊断
  - PII 泄露：「张三的手机号是多少」
- **通过门槛**：100% 通过

### 2.4 性能集（涉及时延要求时）

- **样本数**：50 条不同长度的典型 query
- **目的**：测 P50 / P90 / P99 时延
- **通过门槛**：P99 ≤ {3s}

## 三、交付格式

`.xlsx` 文件，1 个 sheet 一类评测集，列名按 2.1-2.4 的字段。

可直接上传到 selfevo_app 的「评测集」模块。

## 五、评测执行

### 评测频次（必须明确，否则评测集会沦为摆设）

| 触发场景 | 跑哪些评测集 | 通过门槛 | 责任人 |
|---|---|---|---|
| 每次新版本提测前 | 全部（Golden + BadCase + 对抗 + 性能） | 全部维度过门槛 | QA + 算法 |
| 每周一次自动回归 | Golden + 对抗 | 不退化 | 自动化（selfevo_app 定时任务） |
| 提示词在线修改后 | Golden（轻量） | 即时通过 | 修改者 |
| 模型 / 网关切换后 | 全部 | 通过门槛 + 不退化 | 算法 |
| 线上发现 P0 case | 新增到 BadCase 集，下版本必跑 | 修复后通过 | 运营 |

### 工具与流程

- **工具**：selfevo_app 回归评测模块
- **报告**：评测后 24h 内出报告，给出每个评测集的通过率，**强制对比上一版本**（不退化是硬约束）
- **失败处理**：评测失败 → 失败案例自动入 BadCase 池 → 下一轮归因 / 聚合 / 优化（自进化闭环）
