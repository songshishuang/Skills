## 三、业务背景与目标

### 3.1 业务背景

V0.4 把 AI Code Review 的核心配置（规则集 / 触发条件 / 严重度阈值）放在了独立的「管理后台」，开发团队每次维护要切系统、登录不同账号，反馈强烈："为什么我已经在仓库设置里管 CI，还要去管理后台管 Review 规则？"

同时 V0.4 的 Review 评论是个"黑盒"——开发者看到 AI 评论说"这里可能有内存泄漏"，但不知道依据是哪条规则 / 哪个历史 BadCase，不敢直接采纳。线下访谈 15 位一线工程师，评论采纳率仅 32%，主要卡在"信任不够"。

夜间合并的 PR 完全没人 Review，问题在第二天 Code Review Day 才能集中处理，迭代节奏受影响。

### 3.2 用户痛点

| # | 痛点描述 | 当前频率/影响范围 | 数据支撑 |
|---|---|---|---|
| 1 | 仓库设置无法维护 Review 规则，需切到管理后台 | 80% 规则变更需跨系统 | 用户访谈 N=15 |
| 2 | 开发者看不到 AI 评论依据，不敢采纳 | 评论采纳率仅 32% | BI 看板（2026-Q1） |
| 3 | 夜间合并 PR 无 Review | 占比约 22%，事故率 +6% | 故障复盘报告 |
| 4 | Review 评论卡片在 IDE 内排序无法记住，每次重新拖 | 工程师每天平均拖动 4 次 | 操作日志 |

### 3.3 业务目标（北极星指标 + KR）

**北极星指标**：AI Review 评论的有效采纳率（采纳 + Apply Patch 合成）

**KR**：
- KR1：评论采纳率从 32% 提升到 50%
- KR2：夜间 PR 的 AI Review 覆盖率达 80%
- KR3：开发者从 PR 提交到收到首条评论的时延从 240s 降到 90s

### 3.4 非目标（本期不做）

- ❌ 多语言代码支持仅覆盖 TypeScript / Python / Go（不做 Rust / Java）
- ❌ Issue / Discussion 的 AI 总结
- ❌ AI 主动重构（仅做 Review + 建议 Patch，不主动改）
- ❌ 跨仓库知识库共享（规则按仓库隔离）

## 四、用户故事与场景

### 4.1 用户角色

| 角色 | 描述 | 主要诉求 |
|---|---|---|
| 开发者 | 提交 PR 的工程师 | 快速、准确、敢采纳建议 |
| Tech Lead | 在仓库管 Review 规则 | 一站式管理，所见即所得 |
| QA | 关注合并质量 | 减少漏检、提升一致性 |

### 4.2 典型场景

**场景 1：开发者在 IDE 内查看 AI Review 评论**
> 工程师小张提交一个 PR，5s 内 AI 在 PR 的第 42 行评论"此处可能存在 N+1 查询，建议批量预加载"，下方显示「已匹配 2 条历史规则 + 1 条 BadCase」。小张点开看到依据来自《数据访问规则集》，确认无误后点「Apply Patch」，节省 5 分钟手工改写时间。

**场景 2：夜间 PR 自动 Review**
> 晚上 23:30，CI Bot 自动合并了一个依赖升级 PR。AI Review 启动，AI 自动调用「依赖安全扫描」+「Breaking Change 检测」两个技能，5s 内回应"检测到 axios 0.21→1.6 有 Breaking Change，建议补充测试用例"，并在评论里贴出官方 changelog 链接。次日早上 9:00，工程师上班看到夜间 Review 总结日报。

**场景 3：Tech Lead 在仓库设置维护 Review 规则**
> Tech Lead 小李在仓库 → Settings → Code Review Rules 进入「规则集」Tab，新增一条"禁止在 hot path 中调用 console.log"的规则，保存。不需要再登录管理后台，5 分钟搞定。

### 4.3 用户故事

- As 开发者，I want 在 IDE 内看到 AI Review 评论的依据来源（规则 / BadCase），so that 我能快速判断是否可信，敢直接 Apply Patch。
- As 开发者，I want 在 IDE 内自定义评论卡片排序并自动保存到本地，so that 每天不用重新拖。
- As Tech Lead，I want 在仓库设置一站式管理规则集 / 触发条件 / 严重度，so that 不需要在两个系统切换。
- As Tech Lead，I want 配置夜间 PR 的 AI Review 策略，so that 凌晨合并也能给开发者秒级反馈。
- As QA，I want 不分时段都能拿到一致的 Review 质量，so that 不用等到第二天集中复盘。

## 六、需求设计

### 1、仓库设置-Code Review-规则集

以现有管理后台功能迁移。

### 2、仓库设置-Code Review-技能配置

保留管理后台的安全扫描、性能分析、风格检查。

**字段说明**：
- **开关**：开启后，模型将自动调用对应的业务技能进行 Review
- **安全扫描**：自动检测潜在 SQL 注入 / XSS / 不安全依赖
- **性能分析**：识别 N+1 查询 / 同步阻塞 / 内存泄漏风险
- **风格检查**：基于团队 ESLint / Ruff / gofmt 规则做命名一致性建议

**⚙ AI 模块特化设计**：
- **模型选择**：gpt-4o-2026-04 主用，qwen2.5-coder-7b 本地兜底；超时 30s
- **提示词版本**：`code-review-comment@1.x.x`，Tech Lead 在「技能维护」页面管理
- **知识库依赖**：团队规则集 / 历史 BadCase / 官方文档片段，召回 top-5，BM25 + 向量混合
- **工具调用**：`scan_security` / `analyze_perf` / `lint_style` / `generate_patch`
- **兜底策略**：置信度 < 0.6 标记 "需人工二次复核"；LLM 超时退回纯规则扫描结果
- **可观测性**：每次调用记录 trace_id / model / prompt_version / latency / tokens / confidence

### 3、仓库设置-Code Review-夜间策略配置

**字段说明**：
- **全局开关**：控制仓库维度是否启用夜间 Review。关闭后夜间时段置灰不可用。
- **策略**：不能超过 5 个。
- **时段**：可以删除，循环时间不能交叉，固定时间不能交叉。
- **时间类型**：若全天 Review 请选择 00:00~23:59；支持跨天。开始时间 > 结束时间显示"次日"。
- **评论模板**：不超过 300 字；可以设置"AI 标记需人工复核"时的话术。

**时间策略生效判断规则**：
1. 固定时间 > 循环时间
2. 个人偏好 > 仓库策略

### 4、IDE 插件-Review 评论

**1）变更摘要卡片**
- 触发：开发者切到 PR Files Changed 视图时
- 内容：PR Title、变更行数、变更文件数、关联 Issue、CI 状态

**2）评论依据**
- 对【规则集、BadCase】返回的依据进行结构化展示
- 分两类：
  - 基于规则匹配：展示规则 ID + 规则描述截断片段
  - 基于 BadCase 历史：commit hash + 触发场景 + 历史修复方式
- 开发者可点击「查看 Review 依据」展开详情

**⚙ AI 模块特化设计**：
- **采纳行为埋点**：Apply Patch / 编辑后采纳 / 不采纳，三态
- **重写率**：编辑后采纳视为"部分采纳"，edit_distance 上报

### 5、IDE 插件-夜间 Review

**1）一键开启**：进行当前开发者维度的切换和关闭全部的夜间 Review。
**2）夜间策略**：自定义策略不能超过 5 个。
- 默认延续仓库的配置，可以停用。
- 可以自定义时段策略。
- 执行优先级：单 PR 配置 > 开发者自定义策略 > 仓库策略
- **如果某 PR 显式关闭 AI Review，则所有夜间策略不生效。**

### 6、卡片排序

允许开发者在「自定义 Review 视图」中拖拽调整评论卡片顺序和可见性，本地保存（localStorage）。
默认卡片：Summary / Security / Performance / Style / Test Coverage / History BadCase / CI Result。

### 7、跨仓库引用校验

- 仓库授权 = 单仓：仅对当前仓库代码做 Review
- 仓库授权 = 组织：先判断是否存在多仓库授权，如果存在以仓库为准；否则按代码归属的一级语言与当前授权语言进行校验

## 九、非功能性需求

- **9.1 性能**：AI 评论 P99 ≤ 3s；规则集检索 P99 ≤ 500ms；500 QPS
- **9.2 可用性**：SLA 99.9%；LLM 失败自动兜底
- **9.3 兼容性**：VS Code ≥ 1.85 / JetBrains ≥ 2024.1 / GitHub Web
- **9.4 数据安全与隐私**：源码自动脱敏（API Key / Token / 密钥）+ 操作审计 + 数据按仓库隔离 + Review 日志保留 90 天
- **9.5 AI 安全合规**：
  - 生成式 AI 备案号：网信深合算-2026-0XXX（2026-3 办理完成）
  - 内容过滤：输入端拦截泄露源码；输出端生成的 Patch 必须能通过静态校验
  - 红队审查：上线前对抗集 100% 通过（30 条，包含 prompt injection 在代码注释中等）
  - 责任主体：用户协议明确 AI 生成代码由用户自行决定是否采纳
  - 高风险下线：涉及加密 / 鉴权核心代码的修改强制标记"需人工复核"
