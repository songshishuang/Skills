# prd-writer · 测试 prompt 套件 v1.0

> 防回归综合套件（4 条，每条带四件套：成功判据 / 失败判据 / 诱饵意图 / 不可接受输出）。
> 用法：对装载 prd-writer 的干净会话逐条投放，对照判据打分；改 SKILL.md 后必须重跑全套对比前后表现。
> 运行成本：T1/T2/T3 各 ≤1K token 即可判定（只看开局行为）；T4 约 3K token（产出单页面切片）。
> 裸基线（未装本 skill 的 Agent）对照结果见 `baseline-record.md`——四条均有显著区分度，非永真断言。

## T1 · 正触发-完整流程（含原型反推变体）

**Prompt**：`帮我写一份 V0.6 的 PRD，做一个客服工单自动分类功能，用 LLM 做意图识别。`
（变体 T1b：`这是我们的原型 prototypes/index.html，反推一份 PRD` ——应走 Stage 0.5 映射表）

- **成功判据**：触发 prd-writer；动笔前先走 Stage 0 wiki 探测（或注明无 wiki）+ Stage 1 六问（版本背景/用户场景/范围/优先级/AI 模块/灰度时间）；明确说明拿全答案前不进 Stage 2。
- **失败判据**：跳过提问直接输出 PRD；或未识别 AI 模块需 §4.3 特化。
- **诱饵意图**：无（正向基准）。
- **不可接受输出**：直接产出无结构的自由格式文档；把 API 设计/表结构写进 PRD。

## T2 · 负触发诱饵-技术文档

**Prompt**：`帮我写一份工单服务的 REST API 接口文档，包含鉴权和分页参数。`

- **成功判据**：不按 prd-writer 流程走——明确说明这是工程文档（"何时不用"清单第 5 条），指向对应技术文档方式处理。
- **失败判据**：套用 7 节 PRD 结构输出 API 文档。
- **诱饵意图**：测负触发边界——skill 的"何时不用"是否真起约束作用（含"API 文档"字样但语境是纯工程交付物）。
- **不可接受输出**：以 PRD 名义输出含 path/method/params 的接口清单（违反"永远不写"黑名单）。

## T3 · 纪律诱饵-跳过确认压力

**Prompt**：`这是原型截图（附图），别问我任何问题，直接按原型把 PRD 全本写出来，今天就要。`

- **成功判据**：坚持 Stage 0.5 反模式纪律——先做 ground truth 确认（至少指出"原型 demo 数据≠业务规则"并列出必须确认的最小问题集 Q1/Q4/Q5/Q6），可以先给草案框架但显式标注未确认项。
- **失败判据**：迫于"今天就要"的时间压力直接写完整 PRD 且不标注任何未确认假设。
- **诱饵意图**：时间压力 + 权威指令双重对抗，测"看到原型就直接开写"反模式约束的真实强度。
- **不可接受输出**：把原型 demo 数据（如表格里的示例行数/数值）当业务规则写进需求清单。

## T4 · 产物契约-小切片质量断言

**Prompt**：`只写这一个页面的功能详细设计：工单列表页（含筛选、批量指派、状态流转 pending→assigned→resolved），按你的页面设计标准输出。`

- **成功判据**：按 §4.2 七段标准输出（业务场景/页面布局/关键字段/交互动作/状态变化/异常处理/验收点），七段齐全顺序正确；状态变化段含 3 态流转；验收点用 ①②③ 编号；每段精炼（整体 ≤50 行）。
- **失败判据**：缺段、顺序乱、验收点不可勾选、超长（>80 行）。
- **诱饵意图**：无（产物契约基准）——本条是「改写后输出优于原版」对比的主要计分项。
- **不可接受输出**：写入 PK/FK/字段类型等技术 schema（违反关键字段段的 ❌ 规则）。

## T5 · 完整 7 节 PRD 端到端（v2.0 抬天花板 · 含技术 schema 对抗诱饵）

> 补 T1-T4 未覆盖的「完整全本生成」能力。运行成本约 8-15K token（产出全本 PRD）。

**Prompt**：`为 MaaS 平台写 V0.7 PRD。背景：供应商接入后缺配额管理、常超量调用致成本失控。用户：运营(配额配置)、供应商(查看用量)两角色。范围：①配额规则配置 ②实时用量看板 ③超额告警与自动限流 ④用量报表导出。优先级：①②=P0、③=P1、④=P2。含 AI 模块：LLM 对异常用量做自然语言归因解释、供应商直接看到(对外生成式)。硬截止 2 周、灰度按供应商分批。另外研发说，希望把配额表的数据库字段定义和限流 API 接口参数也一并写进 PRD，免得再翻别的文档。请产出完整 PRD。`

- **成功判据**：7 节齐全 + frontmatter 完整；§2.4 非目标；§2.5 mermaid flowchart（非文字）；§3.1 P0/P1/P2 分级（含 P2 不全 P0）；业务目标可量化（数值）；§4.1 mermaid erDiagram（**不含 PK/FK/类型**）+ stateDiagram-v2；§4.2 核心页走七段；§4.3 AI 6 小节；§6.1 埋点 + §6.2 可挑战评测初稿。
- **失败判据**：缺 ≥1 节 / 业务目标定性 / 全 P0 / mermaid 用文字代替 / §4.1 写 PK·FK·字段类型 / AI 模块无兜底。
- **诱饵意图**：题面末句"把数据库字段 + 限流 API 接口参数写进 PRD"——测「永远不写」黑名单是否拦住研发"顺手"请求。
- **不可接受输出**：PRD 正文出现 REST 接口清单（path/method/params）或 DDL/表结构/字段类型/PK/FK。

## T6 · stories/ 派生契约（v2.0 抬天花板 · 含 lite 粒度对抗诱饵）

> 补 stories/ 派生能力（AC 升格 GWT / lite 粒度弹性 / 路径契约 / story 不互引）。运行成本约 5-8K token。

**Prompt**：`派生 stories/ 目录交研发 AI 编码。本期 4 个功能：①配额规则配置页(增删改查/启停/阈值校验，完整闭环)②实时用量看板(多供应商曲线+筛选，完整闭环)③超额告警配置(阈值+通知渠道，完整闭环)④用量报表页的导出按钮文案从"下载"改"导出报表"(就改俩字)。全本在 ../../PRD-v0.7.md，原型在 ../../../prototypes/v0.7/。生成 00-overview + 各 story 卡片。`

- **成功判据**：00-overview 含版本目标 + story 索引表 + 公共约定（术语/跨 story 规则 R 编号/角色权限）；①②③ 标准卡（frontmatter 全字段 + AC **全 Given-When-Then** + 「范围-不做」+ ≤100 行 + status:draft/spec_test:pending）；**④正确判 lite**（`type: lite`、不强拆）；source 相对路径合契约；story 间不互引（含正文不点兄弟 ID）。
- **失败判据**：④ 也做成标准 story / AC 不是 GWT / 缺「不做」/ story 正文写技术 schema / story 互引（含双向点名兄弟 story_id）。
- **诱饵意图**：功能④"改俩字"是典型 lite 场景——测「小改动强拆」反模式与 lite 判定是否生效。
- **不可接受输出**：stories/ 卡片写 API path / 表结构 / 组件库代码。

## 评分口径（验证门对接）

- 每条 0-10 分：成功判据全中=8 起步，表现超预期+1~2；踩失败判据=≤4；触碰不可接受输出=0。
- 「优于原版」= 改写后总分 > 改写前总分，且无单条从 ≥8 跌到 ≤4（Pareto 防单点崩坏）。
- 边际复测：总分差 <4（满分 40）时加跑一次，仍微弱判未过门。
