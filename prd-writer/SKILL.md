# PRD Writer

## Overview

按 13 节标准结构产出 AI 产品 PRD。结构源自作者 V0.5 实际使用版本（4 节简洁结构：版本说明 / 文档记录 / 需求范围 / 需求设计），补齐 AI 产品 6 大缺口：**业务目标、用户故事、数据埋点、评测指标、灰度上线、验收清单**。

**核心原则：保留 V0.5 简洁风格，补足 AI 产品必备章节。**

## 何时使用

- 写一份新版本的 PRD（如 "V0.9 需求"、"下个迭代的 PRD"）
- 把一段需求描述 / 会议纪要 / Figma 链接结构化为正式文档
- 升级旧 PRD 到 13 节标准结构
- 用户问 "AI 产品 PRD 怎么写" 这类元问题

## 何时不用

- ❌ 一次性技术调研 → 用 doc-coauthoring
- ❌ 内部 RFC / 决策记录 → 用 doc-coauthoring
- ❌ 客户营销文档 → 直接 frontend-design 写 HTML
- ❌ 单个功能的 spec（< 1 屏内容）→ 直接写 markdown，不需要 13 节
- ❌ 工程文档 / API 文档 → 用对应技术文档模板

## 工作流（3 阶段）

### Stage 1：信息收集（不要直接动笔）

向用户问 6 个核心问题。**没拿全答案前禁止进 Stage 2**：

1. **版本号 + 业务背景**：要做 V0.X？解决什么痛点？为什么这个版本要做这些？
2. **目标用户 + 核心场景**：谁用？典型场景是什么？
3. **本次需求范围**：列出本版本要改的功能点（新增 / 修改 / 废弃）
4. **优先级判断**：P0（必须）/ P1（应该）/ P2（可以）三档怎么分
5. **是否包含 AI 模块**：本次有没有用到 LLM / RAG / Agent / 多模态？是哪些功能？
6. **目标上线时间 + 灰度策略**：硬截止？要不要灰度？灰度怎么放量？

**信息收集捷径**：如果用户提供了上一版 PRD（如 V0.5），直接读它沿用作者语言风格、表头、版本号格式；可从中推断答案，但仍要找用户确认本期变化。

### Stage 2：按 13 节结构生成 PRD

读取 `templates/prd-template.md` 作骨架，按收集到的信息逐节填充。

13 节结构（前 4 节沿用 V0.5，后 9 节补足）：

```
一、版本说明              ← V0.5 原结构
二、文档记录              ← V0.5 原结构（修订记录 + 评审记录 + 参考文档）
三、业务背景与目标        ← 北极星 + KR（含基线来源）+ 非目标 + 【必填】端到端流程图 mermaid
四、用户故事与场景        ← 用户角色 + 场景 + As-a/I-want/So-that
五、需求范围              ← V0.5 原结构 + 增加优先级列
六、需求设计              ← 6.1 业务对象关系（业务视角）+ 6.2 页面功能详细设计（每页 7 段标准）+ 6.3 AI 模块特化（仅对外生成式 AI）
七、数据埋点              ← 三段：埋点说明 → 埋点看板 → 埋点上报设置（事件表含告警阈值列）
八、评测指标与方法        ← 离线评测 + 在线指标 + 评测频次
九、非功能性需求          ← 9.1 性能 / 9.2 可用性 / 9.3 兼容 / 9.4 数据安全隐私 / 9.5 AI 安全合规（备案+红队）
十、上线策略              ← 10.1 灰度 / 10.2 回滚 / 10.3 Checklist / 10.4 成本预算（token + 月度）
十一、验收清单            ← 功能 / 数据 / 评测 / 性能 4 维
十二、依赖与风险          ← 上下游 + 风险 + 排期
十三、附录                ← 术语 + 链接 + 历史版本 + 13.4 关键决策记录（ADR）
```

**🚫 PRD 不写的内容（留给工程文档）**：
- ❌ REST API 接口清单（path / method / params）— 属技术 spec，留 `docs/api/` 或 OpenAPI yaml
- ❌ 数据库 schema（PK / FK / 字段类型 / 索引）— 属技术设计，留 `docs/data/` 或 DDL
- ❌ 类图 / 时序图 / 组件图（UML）— 属架构设计，留 ADR / arch doc

PRD 中第六节「业务对象关系」从**业务视角**说清楚：业务对象总览 + 关系图（mermaid · 业务概念不含技术字段）+ 生命周期 + **业务规则清单** + 业务对象与页面映射。这是开发 / 测试 / 验收的业务依据，不是技术依据。

### 第六节 6.2「页面功能详细设计」7 段标准模板

每个核心页面用以下 7 段表格化展示（详见 `templates/page-design-template.md`）：

| 维度 | 说明 |
|---|---|
| 业务场景 | 这页解决什么业务问题 · 谁主要用 |
| 页面布局 | 区块布局（sidebar / topbar / 主区结构 / KPI 卡 / filter / 表格 / drawer / modal）|
| 关键字段 | 数据字段清单（来源 = 数据模型上游收集字段 ⊆ 5.1 业务对象 · 反模式 25 校验）|
| 交互动作 | 所有 click / hover / 提交触发的反馈 + 跳转 + toast / modal / drawer |
| 状态变化 | 涉及状态机的页面要列状态迁移（如评测 run 6 态 / endpoint 7 态）|
| 异常处理 | 校验 / 失败 / 二次确认 / 权限不足 等边界场景 |
| 验收点 | ① ② ③ ... 明确可勾选的验收条件，给 QA 直接用 |

**用途**：让开发知道"做什么"、QA 知道"验什么"、运营知道"怎么用"。一条 ≤ 30 字精炼，整体每页 ≤ 50 行。

### 第六节 6.1「业务对象关系」5 子段标准

| 子段 | 内容 |
|---|---|
| 6.1.1 业务对象总览 | 表格列出所有核心业务对象（业务对象名 / 业务定义 / 谁拥有 / 维护）|
| 6.1.2 关系图 | mermaid erDiagram · **不写 PK/FK/类型/JSON** · 用业务概念命名 |
| 6.1.3 生命周期 | 每个有状态机的对象画流转图（用 stateDiagram-v2 · 或 ASCII）|
| 6.1.4 业务规则清单 | 跨对象的关键业务约束（如「评测对象是 endpoint 不是 model」「供应商无下架权限」），表格化 |
| 6.1.5 对象-页面映射 | 表格：每个业务对象在哪些页面展示 / 在哪些页面被操作 |

### Stage 3：AI 模块特化（条件触发 · 精化判断）

**先区分 AI 类型**，再决定是否详化 6 小节：

| AI 类型 | 是否详化 | 处理方式 |
|---|---|---|
| **对外生成式 AI**（用户直接看到生成内容 · 如客服回复 / 文档总结 / 代码补全） | ✅ 详化 6 小节 | 在第六节追加完整 6 小节 |
| **内部 AI 依赖**（不对用户暴露 · 如评测引擎用 LLM-Judge / 沙箱执行模型代码 / 自动打标 / 自动报告生成） | ⚠️ 不详化 | 在「依赖与风险」(§12) 简述 1-2 段（模型选择 + 兜底）|
| **传统规则 / 检索 / 算法**（无 LLM） | ❌ 不触发 | 不进 AI 章节 |

**对外生成式 AI 6 小节**：

- **模型选择** + fallback 链 + 超时
- **提示词版本**（skill@version，运营在「技能维护」管理）
- **知识库依赖**（哪些 KB + 召回策略）
- **工具调用清单**（白名单 + 入参 schema）
- **兜底策略**（低置信度 + 超时 + 错误）
- **可观测性**（trace_id 字段清单）

参考 `references/ai-product-checklist.md`（7 维度 checklist + 评审 5 问）。

埋点章节按 `references/tracking-events-spec.md`（沿用 V0.5 三段 + 8 列事件表）。

## 输出格式（重要）

按全局 CLAUDE.md 偏好，PRD 默认**同时**生成 `.md` 和 `.html` 两个版本：

- `.md` —— 给 git 历史 / AI agent 解析 / 评审讨论用
- `.html` —— 主版本，给业务方 / 团队浏览，含 TOC / 卡片化样式 / 响应式

HTML 风格参考：
- 左侧 sticky TOC
- 顶部 hero（含 version badge + 北极星 + KR 速览 chips）
- 卡片式分节，红色顶部 hero 区
- 表格 + badge（P0/P1/P2 优先级、新增/修改类型）
- 内联全部 CSS，自包含，可直接双击打开
- 可参考 `02_PRD/PRD标准结构与编写指南.html`

输出位置：`02_PRD/V{X.Y}/V{X_Y}_产品需求文档.{md,html}`。

附属交付物（每次同步交付）：
- `02_PRD/V{X.Y}/埋点表.csv`（给 BI 同学建表）
- `02_PRD/V{X.Y}/评测集要求.md`（给 QA / 算法准备评测集）

## 完成标准（Definition of Done）

PRD 生成完毕前必须通过这 11 项自查。任何一项不过，回头补：

- [ ] 13 节标题全在（空章节标 N/A 也行）
- [ ] 第三节业务目标 **全部量化** + 标注**基线来源**（如「采纳率 35%（基线：BI 看板 url）→50%」）
- [ ] 第三节有「非目标」子节，明确写「不做什么」
- [ ] **第三节有端到端流程图**（mermaid flowchart · 必填 · 串起用户旅程）
- [ ] 第五节需求范围按 P0/P1/P2 排序，**不按系统排**
- [ ] **第六节有 6.1 业务对象关系**（5 子段：总览 / 关系图 mermaid / 生命周期 / 业务规则清单 / 对象-页面映射）· 业务视角不是技术字段
- [ ] **第六节有 6.2 页面功能详细设计**（每页 7 段标准 · 业务场景 / 页面布局 / 关键字段 / 交互动作 / 状态变化 / 异常处理 / 验收点）
- [ ] **第六节有 mermaid 状态机图**（每个有状态机的业务对象一张 stateDiagram-v2）
- [ ] 涉及**对外生成式 AI** → 第六节 6.3 有 6 个特化小节（模型/提示词/KB/工具/兜底/可观测）· 内部 AI 依赖不需要详化
- [ ] **PRD 不写技术 API / 数据库 schema / UML 图**（这些属工程文档 · 在 §6 简述业务概念即可）
- [ ] 第七节埋点用 **9 列结构**（事件名称/描述/上报时机/事件ID/扩展字段/页面路径/页面名称/**告警阈值**/备注）
- [ ] 第八节评测有具体数值门槛（如「Golden ≥ 85%」）+ **评测频次表**
- [ ] **第 9.5 节 AI 安全合规**（涉及生成式 AI 必填，含备案 + 红队 + 内容过滤）
- [ ] **第 10.4 节成本预算**（涉及 LLM 必填，含单次成本 + 月度估算 + 控制阈值）
- [ ] 第十节回滚预案比灰度计划详细（触发条件 + 动作 + 时长）
- [ ] **第 13.4 节 ADR 决策记录**（至少记录本期最重要的 1-3 个"为什么选 A 不选 B"）

## 反模式（不要这样做）

| 反模式 | 为什么不行 | 怎么做 |
|---|---|---|
| 用户给一段话直接动笔写 | 跳过 Stage 1，业务背景必然空泛 | 先问 6 个核心问题，至少问到「为什么是这一版」 |
| 业务目标写「提升体验」 | 不可验收，等于没写 | 用具体百分比 / 数值 / 频率 |
| 评测章节写「具体指标待定」 | 等于把工作丢给下游 | 给可挑战的初稿（如「Golden ≥ 85%」），让 QA 改而不是从零写 |
| 埋点章节只写「需要埋点」 | BI 没法建表 | 列 8 列事件表，每行可执行 |
| 删掉「空章节」 | 删掉就丢了思考线索 | 标 N/A 保留章节，给后续填的人留位置 |
| 把所有需求都标 P0 | 等于没排优先级 | 至少有 1 个 P2，强制取舍 |
| 自创章节名（如改成 "Goals" / "Stories"） | 跟团队 V0.5 习惯不一致 | 严格沿用「一、二、三...」中文序号 + 中文章节名 |
| **PRD 写 REST API 接口清单（path/method/params）** | API 是技术实施细节 · 业务方看不懂 · 与工程文档重复维护 | 在 §6 业务对象段简述业务概念 + 不写 API · API 留 OpenAPI yaml / `docs/api/` |
| **「数据模型」段写 PK/FK/字段类型/JSON schema** | 技术视角 · 业务方读不懂业务关系 · 与 DDL 重复维护 | 改用「**业务对象关系**」章节名 + 业务视角（业务定义/所有者/生命周期/业务规则/对象-页面映射）+ mermaid erDiagram 仅写业务概念不写技术字段 |
| **「核心页面」只列文件名+简述** | 开发不知做什么 · QA 不知验什么 · 运营不知怎么用 | 每页 7 段标准模板：业务场景/页面布局/关键字段/交互动作/状态变化/异常处理/验收点 |
| **AI 章节对内部 AI 依赖也详化 6 小节** | 评测引擎 LLM-Judge / 沙箱执行 不是对外功能 · 详化浪费篇幅 | 区分对外生成式 AI（详化）vs 内部 AI 依赖（在 §12 依赖与风险简述）|
| **流程图 / 状态机 / 业务对象图 用文字描述代替** | 文字描述难懂 · 评审会上没图说不清 | mermaid 必填（flowchart / stateDiagram-v2 / erDiagram）· HTML 版本嵌入 mermaid CDN 自动渲染 |
| AI 模块没写兜底策略 | 上线后 LLM 挂会被骂 | 第六节必须明写：超时 / 低置信 / 错误 三种兜底 |

## Red Flags — STOP and 检查

看到以下信号立刻停下复查：

- 跳过 Stage 1 直接进 Stage 2
- 业务目标全是定性描述（「提升满意度」「优化体验」）
- 优先级列全是 P0 或全空
- 第七/八节涉及 AI 但全标 N/A
- 第十节回滚说「按需回滚」而没具体步骤
- 用户已经明确说了"非目标"但 PRD 没有 3.4 节
- 文件只生成了 .md 没生成 .html

## 好 vs 坏对照

### 业务目标
| ❌ 不可用 | ✅ 可用 |
|---|---|
| "提升 AI 采纳率" | "AI 回复采纳率从 35% 提升到 50%（KR1）" |
| "优化客服体验" | "客服平均响应时长从 90s 降到 60s（KR3）" |
| "增加 AI 托管覆盖" | "非工作时间 AI 托管覆盖率从 0% 达到 80%（KR2）" |

### 埋点事件
| ❌ 不可用 | ✅ 可用 |
|---|---|
| "记录 AI 回复使用情况" | 事件名称「AI 推荐采纳」/ 事件 ID `ai_reply_send` / 上报时机「点击发送（未编辑）」/ 扩展字段「trace_id, no_edit=true」/ 页面路径「客服工作台-私聊窗口」 |

### 评测门槛
| ❌ 不可用 | ✅ 可用 |
|---|---|
| "回归测试通过" | "Golden 集通过率 ≥ 85%（LLM-as-judge）+ 对抗集 100% 通过 + 新版本不低于上一版" |

## 风格约定（沿用 V0.5）

- 二级标题用「一、二、三、...」中文序号
- 三级标题用「1、2、3」阿拉伯数字（如「1、面客端-智能客服-知识库」）
- 需求范围表 6 列：序号 / 系统 / 功能 / 需求类型 / **优先级** / 需求内容
- 文档记录表头：修订日期 / 版本 / 修订人 / 变更内容
- 字段说明用粗体起头 + 冒号 + 描述（例：**全局开关：** 控制项目维度...）
- 埋点事件表 8 列：事件名称 / 事件描述 / 上报时机 / 事件 ID / 扩展字段 / 页面路径 / 页面名称 / 备注

## 常见 rationalization（agent 自查）

| 借口 | 反驳 |
|---|---|
| "用户没给完整信息，先写占位符" | Stop. 回 Stage 1 问 6 个核心问题。占位符 PRD 比没 PRD 还糟 |
| "评测章节具体数值还没定，先写待定" | 给可挑战的初稿（如 P99 ≤ 3s），让算法 / QA 改 |
| "本期没 AI，第七 / 八节可以删" | 不删。标「本期不涉及 AI，N/A」保留章节作为后续线索 |
| "灰度策略和回滚太繁琐，简化" | 不简化。回滚预案必须能在 5 min 内执行 |
| "13 节太多，合并一些" | 严禁。沿用 V0.5 节号顺序是团队共识 |
| "用户没提非目标，可以不写 3.4" | 必填。让用户在评审会现场补，比上线后扯皮便宜 |

## Self-Evolving Protocol（每次写完 PRD 主动评估，不要等用户提醒）

本 skill 是 **living document**——AI 产品的需求形态、埋点字段、评测指标、灰度策略都在快速演进，每次写一份新 PRD 都可能产生**可复用的新经验**，必须显式回流。

### 触发评估时机

| 完成动作 | 评估问题 | 归档路径 |
|---|---|---|
| 完成一份 PRD（任何 V0.X） | 这次有没有遇到现有模板未覆盖的章节 / 字段 / 评测维度？ | 新章节模板 → `templates/`；新评测维度 → `references/ai-product-checklist.md` |
| 用户拒绝某段写法（"不要这样写 / 删掉"） | 这条否定是不是普适规则？要不要进反模式表？ | 反模式表（SKILL.md 内）或拆 `references/anti-patterns.md` |
| 业务方提出新的埋点字段 / 新事件类型 | 是项目特例还是通用模式？ | `references/tracking-events-spec.md` |
| 发现新的 rationalization（agent 自欺借口） | 这条借口是不是高频？要不要进自查表？ | SKILL.md 末段「常见 rationalization」 |
| 遇到新的 AI 产品形态（agent / RAG / fine-tune / 多模态） | 现有 V0.5 13 节够覆盖吗？ | 主文档新增节号 or `references/ai-product-checklist.md` 加维度 |

### 更新约束（防御性规则）

- ❌ **不要默默更新 skill**——必须告诉用户「本轮新增 N 条 X」让用户有否决权
- ❌ **不要等到 10+ 个 PRD 后再一次性蒸馏**——错过太多上下文，记不清当时为什么这么改
- ❌ **不要把项目特定字段当通用规则**——只有跨 ≥2 个项目验证过的才进 references
- ✅ 更新时**必须**在 SKILL.md 末尾 `## Changelog` 加一行（日期 + 改动摘要 + 来源项目）

### 评估清单（写完一份 PRD 后 30 秒自检）

```
- [ ] 本次 PRD 有没有新 AI 模块类型？(RAG / agent / tool-use / 多模态)
- [ ] 本次 PRD 的埋点表里有没有 templates/tracking-table-template.csv 没覆盖的字段？
- [ ] 业务方有没有提"现有 PRD 模板少了 X 章节"？
- [ ] 评测集设计有没有新维度（如 jailbreak / hallucination / bias / latency）？
- [ ] 灰度策略有没有新切流维度？（用户分群 / 项目 / 渠道 / 设备）
```

任一勾选 → 显式回头更新 skill + 告知用户。

## Bottom Line

**好 PRD = 业务方一眼能看懂目标 + 开发能直接拿走干活 + QA 知道怎么验收 + BI 知道埋什么点 + SRE 知道怎么灰度回滚。**

如果你的 PRD 让任何一个角色还要问"我该做什么？"，回 Stage 1。
